# Milestone v1.0: Codex Mobile MVP

**Status:** Shipped 2026-04-18
**Archived:** 2026-04-19
**Phases:** 1-5 (+ inserted 01.1 hotfix)
**Total Plans:** 21

## Overview

v1.0 shipped the core Codex Mobile product: secure QR pairing and durable device sessions, an outbound-only local bridge wired to `codex app-server`, a phone-first live control surface, durable audit and revoke flows, and multi-instance relay ownership plus Fly replay hardening.

The implementation milestone is complete, but the archive intentionally records accepted verification debt. The shipped code covers the planned product surface; what remains deferred is manual Fly validation and several missing milestone-level verification artifacts. See [v1.0-MILESTONE-AUDIT.md](./v1.0-MILESTONE-AUDIT.md).

## Phases

### Phase 1: Identity & Pairing Foundation

**Goal:** Establish a secure internet-facing entry point for Codex Mobile with QR-based pairing and 7-day device sessions.
**Depends on:** Nothing (first phase)
**Requirements:** AUTH-01, AUTH-02, PAIR-01, PAIR-02, PAIR-03, PAIR-04, PAIR-05, SEC-01, SEC-06, OPS-01
**Plans:** 7 plans

**Success criteria:**
- User can sign into the web app and begin pairing from a phone browser.
- Local terminal can display a short-lived QR code and fallback code for pairing.
- Terminal and browser both show a verification phrase before a 7-day device session is granted.
- Pairing and web access work over Fly-hosted services without any inbound port on the developer machine.

Plans:

- [x] 01-01: Define trust boundaries, shared protocol contracts, and auth/session model
- [x] 01-02: Implement web auth, pairing APIs, terminal QR flow, and confirmation UX
- [x] 01-03: Deploy baseline web/control services to Fly.io with TLS, secrets, and health checks
- [x] 01-04: Fix middleware CR-01 and add missing `GET /api/pairings/[pairingId]` handler
- [x] 01-05: Add pairing token bearer verification, Origin/CSRF protection, rate limits, waitForRedeem error propagation, and 32-byte cookie-secret enforcement
- [x] 01-06: Harden Fly deploy workflow and Dockerfiles; document single-machine pairing-store limits
- [x] 01-07: Remove `/confirm` auth drift, cap rate-limit buckets, normalize 500 fallthroughs, and align pairing status schema

### Phase 4: Approval, Audit & Device Safety

**Goal:** Make repeated remote use safe through revocation, reconnect, audit, and clear approval handling.

- The mobile UI can recover from short disconnects without repeating the full pairing flow.

Plans:

### Phase 5: Multi-Instance Routing & Production Hardening

**Depends on:** Phase 4
**Requirements:** SEC-04, OPS-02, OPS-03, OPS-04

- Operators can inspect connection health, ownership state, queue pressure, and disconnect reasons.

## Milestone Summary

**Inserted phase:**

- Phase 01.1 was inserted after Phase 1 when verification surfaced a real browser-cookie delivery gap in the pairing flow.

**Key accomplishments:**

- Shipped secure QR pairing, terminal confirmation, and durable 7-day device sessions through the hosted web app and local bridge.
- Integrated the outbound-only bridge with `codex app-server` over stdio so remote users can attach to real local Codex sessions without losing conversation context or approval semantics.
- Delivered a mobile-first live control surface with structured activity rendering, prompt/steer/interrupt controls, reconnect handling, and explicit terminal session states.
- Added device management, revocation, and append-only audit capture across the web and relay trust boundaries.
- Added durable relay ownership, Fly wrong-instance replay routing, readiness gating, relay ops visibility, and browser backpressure protection for multi-instance scale.

**Key decisions:**

- `codex app-server` remains the primary local integration surface; `codex exec --json` is still fallback-only.
- Remote activity is modeled as product-owned structured events rather than terminal-byte scraping.
- The local bridge remains outbound-only and never exposes the developer machine directly to the internet.
- Trust depends on authenticated device sessions plus short-lived connection credentials instead of forwarding long-lived browser sessions to the relay.
- Fly routing is based on durable relay ownership and replay instead of a single in-memory coordinator.

**Issues deferred:**

- Phase 5 manual UAT was intentionally paused with seven outstanding staging checks.
- Phases `01.1`, `02`, `03`, and `04` still lack milestone-level `VERIFICATION.md` artifacts.
- Phase `01` and Phase `05` verification artifacts exist but remain `human_needed` rather than `passed`.

**Technical debt accepted at archive time:**

- The v1.0 milestone audit remained `gaps_found`, so the milestone was archived as shipped implementation plus deferred verification work rather than a fully validated release.
- Several v1 requirements are implemented but still recorded as verification debt because manual Fly or cross-phase confirmation is outstanding.
- Relay multi-instance behavior, owner-loss recovery, and degraded ops inspection still need real environment validation before broad rollout.
