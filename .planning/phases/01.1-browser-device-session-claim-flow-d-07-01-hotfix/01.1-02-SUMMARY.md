## Decisions Made

1. **Server+client split architecture:** page.tsx remains a pure server component responsible for auth check, redirect, and redeemPairing SSR. All interactive UI (polling, auto-claim, state transitions, error display) lives in the PairingClaimFlow client component which receives serializable string props.

2. **Polling implementation:** Uses setInterval inside useEffect with proper cleanup (clearInterval + clearTimeout). A statusRef tracks current status to prevent stale closure reads in the interval callback. A claimInProgress ref prevents duplicate claim POSTs when the interval fires before the claim fetch completes.

3. **Error state mapping:** Each D-08 error case has a dedicated code path:
   - Network poll errors: silent retry, "Reconnecting..." hint after 2 consecutive failures
   - Claim 403 user_mismatch: hard error with "different account" message, no retry
   - Claim 500: one silent retry, then "Pairing failed" with terminal guidance
   - Timeout: 5-minute wall-clock setTimeout triggers "Pairing timed out"

4. **Test strategy:** Playwright tests verify backend prerequisites (redeemedByUserId persistence, confirmedAt return type, real userId in confirmedByUserId) rather than HTTP-level claim flow, because issueDeviceSession requires a Next.js route handler context that cannot be invoked in-process.
