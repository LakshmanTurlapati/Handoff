# Phase 01.1 Plan 01: Browser Device Session Claim Flow -- Backend Summary

Backend refactor to separate cookie minting from pairing confirmation: added redeemedByUserId to track the redeeming user, refactored confirmPairing into a pure state transition, created the /claim route as the sole cm_device_session cookie-minting path with cookie-auth + user-match verification.

## Decisions Made

1. **D-09 (cookie minting separation):** confirmPairing in pairing-service.ts no longer imports or calls issueDeviceSession. The issueDeviceSession call now lives exclusively in the /claim route handler. The confirm route returns only the state-transition result (verificationPhrase + confirmedAt).

2. **D-11 (user identity persistence):** redeemPairing now persists `redeemedByUserId: input.userId` in the store.update patch. The /claim handler reads this field to verify the authenticated browser user matches the phone user who redeemed the pairing.

3. **D-12 (sentinel removal):** The `pairing-bearer:<pairingId>` sentinel userId in confirm/route.ts has been replaced with `bridge:<pairingId>`. More importantly, confirmPairing now resolves the real user via `row.redeemedByUserId ?? input.userId`, so audit records and the confirmedByUserId field carry real user identity.

4. **PairingConfirmResponse backward compatibility:** deviceSessionId was made optional (not removed) on the protocol schema so older bridge clients that check for the field don't break. The confirmedAt field was added as optional. The bridge client's safeParse continues to work without code changes.
