# Phase 01.1: Browser Device Session Claim Flow — Context

**Gathered:** 2026-04-11
**Status:** Ready for planning

<domain>
## Phase Boundary

Close the D-07-01 gap discovered during Phase 1 verification iteration 2: the phone browser never receives the `cm_device_session` cookie because `/confirm` is called by the bridge CLI, and the `Set-Cookie` header flows to the wrong process. This phase adds a browser-side claim path so the phone can obtain the cookie after the bridge confirms the pairing.

**Scope anchor:** This is a hotfix phase. No new capabilities beyond cookie delivery. No multi-machine store, no Drizzle migration, no new sign-in providers.

</domain>

<decisions>
## Implementation Decisions

### Delivery Mechanism
- **D-01:** Polling + claim endpoint. Browser polls `GET /api/pairings/[id]` at fixed intervals; when status transitions to `confirmed`, browser POSTs to a new `POST /api/pairings/[id]/claim` endpoint which mints the `cm_device_session` cookie on the browser's response.
- **D-02:** No SSE, no WebSocket. Phase 1 is in-memory / single-machine; streaming infra is unnecessary complexity.

### Claim Endpoint Auth
- **D-03:** Cookie session + user-match. The `/claim` endpoint requires a valid `cm_web_session` cookie AND verifies `session.user.id === pairing.redeemedByUserId`. Only the user who redeemed (phone user) can claim the cookie. Rejects with 403 + "This pairing was started from a different account" on mismatch.
- **D-04:** Claim response is `Set-Cookie` header (writes `cm_device_session`) plus a JSON success payload: `{ status: "confirmed", deviceSessionId: "<id>", label?: "<device-label>" }`.

### Browser UX
- **D-05:** Auto-claim on detection. When polling detects `status=confirmed`, the browser immediately POSTs `/claim` without any user tap. The user already consented by typing Y in the terminal; a second tap is redundant.
- **D-06:** After successful claim, stay on `/pair/[id]` showing a "This device is paired for 7 days" confirmation. No redirect. User closes the tab or navigates manually when ready.

### Polling Cadence + Error UX
- **D-07:** 2-second fixed polling interval. Stop conditions: state reaches `confirmed` (after claim), `expired`, or `cancelled`; OR a hard wall-clock timeout matching the 5-minute pairing TTL.
- **D-08:** Minimal + actionable error states:
  - Network errors during polling: silent retry, show small "Reconnecting..." hint after 2 consecutive failures.
  - Claim 500 (`internal_error`): retry once silently, then show "Pairing failed. Start a new pairing from the terminal."
  - Claim 403 user-mismatch: hard error "This pairing was started from a different account. Sign in as [hint] or start a new pairing."
  - Pairing expired: "Pairing expired. Start a new pairing from the terminal and try again." (reuses existing STATUS_COPY.expired).

### Confirm Route Cleanup
- **D-09:** Drop `issueDeviceSession` from `/confirm` entirely. `confirmPairing` becomes a pure state-transition function (no cookie minting). The bridge no longer receives any Set-Cookie header. The `/claim` endpoint is the sole cookie-minting path.
- **D-10:** `confirmPairing` return type changes: no longer returns `IssuedDeviceSession`. Returns only `{ pairingId, verificationPhrase, confirmedAt }` (the state transition result). Bridge `pairing-client.ts` response schema updates to match.

### User Identity Persistence
- **D-11:** Add `redeemedByUserId: string | null` field to the in-memory pairing row model (and to `pairing_sessions` Drizzle schema for forward compatibility). Populated by `redeemPairing` when the phone browser server-component calls it. Read by the `/claim` endpoint to verify user-match (D-03).
- **D-12:** `confirmedByUserId` on the pairing row continues to be populated during `confirmPairing`, but now uses the real `pairing.redeemedByUserId` from the row (looked up server-side) instead of the sentinel `pairing-bearer:<id>` from Phase 1's workaround. The sentinel logic in `/confirm/route.ts` is removed.

### Claude's Discretion
- Server vs client component split for the pair page polling (server component renders initial state; client component handles polling + auto-claim + state transitions)
- Schema column placement and naming conventions in `packages/db/src/schema.ts`
- Claim endpoint file location under `apps/web/app/api/pairings/[pairingId]/claim/route.ts`
- Bridge `PairingConfirmResponse` schema updates (optional deviceSession removal) — backwards-compatible approach preferred
- Test structure and mocking strategy for the claim flow in `auth-pairing.spec.ts`

</decisions>

- `PAIRING_AUDIT_EVENTS` — add `claimed` event type for audit trail

### Established Patterns
- Route handlers in `apps/web/app/api/pairings/[pairingId]/` follow App Router conventions: `export async function POST(request, context)`, `export const runtime = "nodejs"`, `export const dynamic = "force-dynamic"`
- Same-origin CSRF guard pattern from `/confirm` and `/redeem` (check `origin` vs `host` header)
- `z.safeParse()` for input validation + `NextResponse.json()` for all responses
- Error handling: specific status-code branches (404, 403, 409, 400) followed by generic `{ error: "internal_error" }` 500 fallthrough

### Integration Points
- Pair page (`apps/web/app/pair/[pairingId]/page.tsx`) is currently a **server component** with no client-side interactivity. Must extract polling + auto-claim into a client component (`"use client"` directive) while keeping initial SSR render.
- Middleware (`apps/web/middleware.ts`) does NOT need an update — the new `/claim` endpoint is POST-only and requires cookie auth, so it should remain behind the existing auth middleware (NOT added to `UNAUTHENTICATED_API_POST_PATHS`).

<specifics>
## Specific Ideas

- The phone pair page should feel like a seamless one-screen flow: user opens QR, signs in, sees "Waiting for confirmation", then ~2 seconds after terminal confirm the screen transitions to "This device is paired for 7 days" without any tap.
- Error messages should be actionable ("Start a new pairing from the terminal") rather than technical ("Error: session mismatch").
- The "Reconnecting..." hint for network errors should be subtle (small text, not a blocking banner).

</specifics>

<deferred>
## Deferred Ideas

- **SSE / WebSocket delivery** — polling is sufficient for Phase 1; consider SSE for Phase 3+ when multi-machine routing needs real-time push
- **Multi-machine claim race** — if two browser tabs claim simultaneously (unlikely but possible), the in-memory store handles it by first-come-first-serve since both calls happen in the same Node process. Drizzle migration will need a row-level lock or `UPDATE ... RETURNING` atomic claim.
- **D-07-02 from Phase 1** — This phase's D-12 decision (use real redeemedByUserId from the row) resolves the sentinel userId problem. The Drizzle FK compatibility is addressed by adding a proper column to the schema.
- **D-07-03 through D-07-07** from Phase 1 — remain deferred as documented

</deferred>
