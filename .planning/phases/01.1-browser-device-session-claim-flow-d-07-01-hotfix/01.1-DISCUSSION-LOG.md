## Delivery Mechanism

| Option | Description | Selected |
|--------|-------------|----------|
| Polling + claim endpoint | Browser polls GET status, detects confirmed, POSTs /claim which mints cookie | Y |
| SSE stream | Browser holds EventSource; server pushes confirmed event | |
| WebSocket | Full duplex for realtime notification | |

**User's choice:** Polling + claim endpoint
**Notes:** Phase 1 is in-memory/single-machine. SSE adds streaming infra and Fly keep-alive concerns. WebSocket is overkill for one-way notification.

### Follow-up: Claim auth

| Option | Description | Selected |
|--------|-------------|----------|
| Cookie session + user-match | Require cm_web_session cookie; verify session.user.id === pairing.redeemedByUserId | Y |
| Cookie session only | Any authenticated user can claim any confirmed pairing | |
| Bearer-gated like /confirm | Not viable -- browser doesn't have the raw token | |

**User's choice:** Cookie session + user-match

### Follow-up: Claim response

| Option | Description | Selected |
|--------|-------------|----------|
| Set-Cookie + JSON success payload | Write Set-Cookie for cm_device_session, return JSON with status, deviceSessionId, label | Y |
| Set-Cookie + 302 redirect | Cookie plus redirect to home | |
| Set-Cookie + 204 No Content | Cookie only, no body | |

**User's choice:** Set-Cookie + JSON success payload

## Browser UX During Claim

## Polling Cadence + Error UX

### Cadence

| Option | Description | Selected |
|--------|-------------|----------|
| 2s fixed | Balance of responsiveness and cost | Y |
| 1s fixed | Most responsive, higher request volume | |
| Exponential backoff (1s to 5s cap) | Smart but more code | |

**User's choice:** 2s fixed

### Error loudness

| Option | Description | Selected |
|--------|-------------|----------|
| Minimal + actionable | Silent retry on network; claim 500 retry once; claim 403 hard error with account hint | Y |

**User's choice:** Minimal + actionable
