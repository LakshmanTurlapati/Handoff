# Phase 01.1: Browser Device Session Claim Flow - Research

**Researched:** 2026-04-10
**Domain:** Next.js App Router cookie issuance, browser polling, server/client component composition
**Confidence:** HIGH

## Summary

This phase closes the D-07-01 gap: the phone browser never receives the `cm_device_session` cookie because `POST /confirm` is called by the bridge CLI, and the `Set-Cookie` header flows to the bridge process (which discards it) rather than to the phone browser. The fix is a new `POST /api/pairings/[id]/claim` endpoint that the browser calls after detecting `status=confirmed` via polling, which mints the cookie on the browser's HTTP response.

The core implementation requires: (1) refactoring `page.tsx` from a pure server component to a server component that renders initial state and delegates to a `"use client"` component for polling + auto-claim, (2) creating the `/claim` route handler that calls `issueDeviceSession` and returns `Set-Cookie`, (3) modifying `redeemPairing` to persist `redeemedByUserId` on the pairing row, (4) modifying `confirmPairing` to stop calling `issueDeviceSession`, and (5) updating the bridge's `PairingConfirmResponse` schema to drop device session fields.

**Primary recommendation:** Build the claim endpoint as a new route handler at `apps/web/app/api/pairings/[pairingId]/claim/route.ts` that requires `cm_web_session` cookie auth (via `auth()`) and verifies `session.user.id === pairing.redeemedByUserId`. The browser client component polls `GET /api/pairings/[id]` at 2s intervals and auto-POSTs to `/claim` when `status=confirmed`.

### Locked Decisions
- **D-01:** Polling + claim endpoint. Browser polls `GET /api/pairings/[id]` at fixed intervals; when status transitions to `confirmed`, browser POSTs to a new `POST /api/pairings/[id]/claim` endpoint which mints the `cm_device_session` cookie on the browser's response.

**What:** The `/claim` route handler calls `issueDeviceSession` which internally calls `(await cookies()).set()` from `next/headers`. This writes the `Set-Cookie` header on the HTTP response that flows back to the browser.
**When to use:** When minting a cookie in a POST route handler that the browser calls directly.

This means the `/claim` endpoint can call `issueDeviceSession` directly and the cookie will be set on the response to the browser. No new cookie-setting code is needed.

### Pattern 3: Polling with useEffect + setInterval
**What:** Client component uses `useEffect` to start a `setInterval` that fetches `GET /api/pairings/[id]` every 2 seconds. When status becomes `confirmed`, it auto-POSTs to `/claim`.
**When to use:** For D-01 polling + auto-claim behavior.
**Example:**
```typescript
useEffect(() => {
  if (terminalStatus) return; // Already in terminal state
  const id = setInterval(async () => {
    try {
      const res = await fetch(`/api/pairings/${pairingId}`);
      if (!res.ok) { handlePollError(res); return; }
      const data = await res.json();
      if (data.status === "confirmed" && !claimed) {
        clearInterval(id);
        await claimDeviceSession();
      } else if (data.status === "expired" || data.status === "cancelled") {
        clearInterval(id);
        setStatus(data.status);
      }
    } catch { handleNetworkError(); }
  }, 2000);
  // Hard wall-clock timeout at 5 minutes
  const timeout = setTimeout(() => { clearInterval(id); setStatus("expired"); }, 5 * 60 * 1000);
  return () => { clearInterval(id); clearTimeout(timeout); };
}, [pairingId, terminalStatus, claimed]);
```

### Anti-Patterns to Avoid
- **Do NOT call `issueDeviceSession` from a server component.** Server components render during SSR and cannot set response headers on the client. Cookie minting must happen in a route handler.
- **Do NOT use `setInterval` without cleanup.** The `useEffect` cleanup function must clear both the interval and the wall-clock timeout to prevent memory leaks on unmount.
- **Do NOT poll after terminal state.** Once status is `confirmed` (and claim succeeds), `expired`, or `cancelled`, stop polling immediately.
- **Do NOT retry claim on 403.** A user-mismatch is permanent; retrying will never succeed. Show the hard error.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Cookie minting | Custom Set-Cookie header construction | `issueDeviceSession()` which calls `cookies().set()` | Already handles JWT signing, cookie attributes (HttpOnly, Secure, SameSite=Lax), 7-day maxAge, hash storage |
| Session auth on /claim | Manual cookie parsing | `auth()` from `apps/web/auth.ts` | Already integrated with Next.js Auth.js, handles JWT verification, session extraction |
| Input validation | Manual field checks | `z.object({}).safeParse()` | Consistent with every other route handler in the codebase |
| CSRF protection | Custom CSRF tokens | Same-origin guard (Origin vs Host) | Already pattern-tested on /confirm and /redeem |
| Polling state machine | Custom event emitter | `useState` + `useEffect` + `setInterval` | React's built-in state management is sufficient for this simple polling loop |

**Key insight:** The `issueDeviceSession` function in `device-session.ts` is the single reusable primitive. It uses `cookies()` from `next/headers`, which works in route handlers (the browser calls `/claim` directly, so the `Set-Cookie` header reaches the browser). No new cookie infrastructure is needed.

## Common Pitfalls

### Pitfall 1: cookies() Only Works in Route Handlers and Server Actions
**What goes wrong:** Attempting to call `cookies().set()` from a server component (like page.tsx) silently fails or throws because server components cannot modify the outgoing response headers.
**Why it happens:** Server components run during rendering, not during request handling. Only route handlers (`route.ts`) and server actions can write response headers.
**How to avoid:** The `/claim` endpoint is a route handler. The browser calls it via `fetch()` from the client component. `issueDeviceSession` (which calls `cookies().set()`) runs inside the route handler context.
**Warning signs:** If you see `issueDeviceSession` being called anywhere other than a route handler or server action, it is wrong.

### Pitfall 2: Polling Race Condition with Claim
**What goes wrong:** The polling interval fires, detects `status=confirmed`, starts the claim fetch... but the next interval also fires before the claim completes, triggering a duplicate claim.
**Why it happens:** `setInterval` does not wait for the async callback to complete.
**How to avoid:** Use a `claimInProgress` ref (`useRef(false)`) that is checked before starting the claim fetch. Set it to `true` before `fetch('/claim')` and never unset it (the claim is a one-shot operation).
**Warning signs:** Multiple `POST /claim` requests appearing in network tab during testing.

### Pitfall 3: Stale Closure in useEffect Polling
**What goes wrong:** The polling callback captures stale state values and makes decisions based on outdated data.
**Why it happens:** JavaScript closures in `setInterval` callbacks capture the state value at the time the effect ran, not the current value.
**How to avoid:** Use `useRef` for mutable state that the interval callback reads (e.g., `claimInProgress`, `currentStatus`). Use `useState` only for values that trigger re-renders (e.g., displayed status, error messages).
**Warning signs:** Polling continues after the component shows "Paired" or "Expired" status.

### Pitfall 4: Middleware Must NOT Block /claim
**What goes wrong:** The new `/claim` endpoint returns 302 redirect to `/sign-in` instead of 200/403.
**Why it happens:** `/claim` requires `cm_web_session` cookie auth, which means it should pass through middleware's `Boolean(auth?.user)` check naturally. But if middleware is accidentally configured to block it, the auth check never reaches the route handler.
**How to avoid:** `/claim` does NOT need to be added to `UNAUTHENTICATED_API_POST_PATHS` or any regex allowlist. It is a browser-only endpoint -- the browser always has `cm_web_session`. The middleware's default `Boolean(auth?.user)` check at line 81 lets authenticated requests through. Confirm this in testing.
**Warning signs:** 302 response on POST /claim during E2E tests.

### Pitfall 5: PairingRow Interface Must Include redeemedByUserId
**What goes wrong:** TypeScript compilation fails because `redeemedByUserId` is not on the `PairingRow` interface.
**Why it happens:** `redeemPairing` tries to pass `redeemedByUserId` in the `store.update()` patch, but the interface does not have that field.
**How to avoid:** Add `redeemedByUserId: string | null` to the `PairingRow` interface first, then add it to the `InMemoryPairingStore`'s initial row shape in `createPairing`, then use it in `redeemPairing`.
**Warning signs:** TypeScript error on `store.update(...)` call.

### Pitfall 6: confirmPairing Return Type Change Breaks Tests
**What goes wrong:** The existing `auth-pairing.spec.ts` test at line 91 calls `confirmPairing(...)` and expects it to resolve (for the expired-pairing case, it expects rejection). Changing the return type from `ConfirmPairingResult` (which includes `deviceSession`) could break tests that inspect the result.
**Why it happens:** The test file calls `confirmPairing` directly (not through HTTP). The test for expired pairings uses `.rejects.toThrow(/expired/)` which does not inspect the result, so it should be safe. But if any test inspects `result.deviceSession`, it will fail.
**How to avoid:** Check all test references to `confirmPairing` result before changing the return type. The current tests only test the rejection path (expired pairing), so the return type change is safe for them.
**Warning signs:** Test failure mentioning `deviceSession` property.

## Open Questions

1. **Claim idempotency: what if the browser double-claims?**
   - What we know: `issueDeviceSession` generates a new device session every time it is called. Two claims would issue two sessions.
   - What's unclear: Should the claim endpoint check if a device session was already issued for this pairing? The in-memory store can check `pairing.claimedAt` or a flag.
   - Recommendation: Add a `claimedAt: Date | null` field to `PairingRow` and check it in `/claim`. If already claimed, return the existing metadata (200) without re-issuing. This prevents duplicate sessions.

2. **Should `/claim` add a new audit event type?**
   - What we know: `PAIRING_AUDIT_EVENTS` currently has `created`, `redeemed`, `confirmed`, `expired`, `confirmFailed`. The CONTEXT.md mentions adding a `claimed` event type.
   - What's unclear: The exact audit metadata shape.
   - Recommendation: Add `claimed: "pairing.claimed"` to `PAIRING_AUDIT_EVENTS` and write an audit row with `{ deviceSessionId, userId, pairingId }`.

3. **Bridge `PairingConfirmResponse` backwards compatibility**
   - What we know: The bridge validates the confirm response against `PairingConfirmResponseSchema` which currently requires `deviceSessionId`. Removing it would break the bridge's `.safeParse()` call.
   - What's unclear: Whether to make `deviceSessionId` optional first and remove later, or remove it now and update the bridge in the same changeset.
   - Recommendation: Update both in the same changeset. Since this is a monorepo and the bridge and web deploy together, there is no backwards-compatibility window. Make `deviceSessionId` optional in the schema and stop sending it from `/confirm`.

### Known Threat Patterns for This Phase

| Pattern | STRIDE | Standard Mitigation |
|---------|--------|---------------------|
| Cross-site POST to /claim | Spoofing/Tampering | Same-origin CSRF guard (Origin vs Host check); SameSite=Lax on cm_web_session cookie |
| Claim by different user who knows pairingId | Elevation of Privilege | D-03: `session.user.id === pairing.redeemedByUserId` check returns 403 on mismatch |
| Duplicate claim issuing multiple device sessions | Information Disclosure | Idempotency guard: check `claimedAt` on pairing row before issuing |
| Polling information disclosure (pairing status visible to anyone with pairingId) | Information Disclosure | Acceptable: pairingId is a UUID4 (128-bit entropy); status response reveals only state, not credentials. Same as existing GET handler. |
| Race condition: claim while confirm still in progress | Tampering | `/claim` checks `status === "confirmed"` before issuing; if the store has not yet transitioned, claim returns 409 |
