# Phase 08.1: Authless Hosted Launch - Context

**Gathered:** 2026-04-20
**Status:** Ready for implementation

<domain>
## Phase Boundary

Replace the hosted GitHub OAuth requirement in the Fly handoff path with a launch-url-native trust model. A valid short-lived `/launch/[publicId]` URL should be able to establish or reuse the durable `cm_device_session`, land the phone on the exact active session that initiated `/handoff`, and keep the existing bridge/relay approval boundary intact.

**Scope anchor:** hosted launch entrypoint, browser principal derivation, device-session issuance/reuse, and documentation/test updates only. This phase does not redesign the local bridge bootstrap, the Codex `/handoff` command surface, or the broader archived v1.0 auth model outside the active handoff flow.

</domain>

<decisions>
## Implementation Decisions

### Hosted Launch Trust Model
- **D-01:** The generated `/launch/[publicId]` URL is the primary browser trust object for this phase; the hosted app must not redirect that flow through GitHub OAuth.
- **D-02:** A valid launch should establish or reuse the durable 7-day `cm_device_session` directly from handoff metadata and the existing bridge installation identity.
- **D-03:** Launches remain short-lived and single-purpose. A revoked or expired handoff must fail closed with a clear hosted error state instead of falling back to a dashboard or generic sign-in page.

### Browser Principal and Scope
- **D-04:** The hosted browser principal is derived from the durable device-session cookie plus its linked bridge installation, not from `auth()` or a separate `cm_web_session`.
- **D-05:** Device/session pages and relay ticket minting should use the bridge-linked device session as the browser identity source so the live flow stays tied to the same local machine that initiated `/handoff`.
- **D-06:** Existing persisted `userId` columns may remain as compatibility substrate, but runtime access control for this phase must no longer require a GitHub-authenticated browser session.

### Launch UX
- **D-07:** Opening a valid launch URL should take the phone directly into the handoff path. If the device is already trusted for that bridge, it should land on the target session without extra setup.
- **D-08:** If the device is not yet trusted, the hosted app should create the durable device session itself and then redirect into the target session rather than surfacing the legacy sign-in page.
- **D-09:** The hosted root and device/session pages should degrade gracefully when no device session exists instead of redirecting to GitHub sign-in.

### Documentation and Runtime Cleanup
- **D-10:** Hosted docs and setup instructions must stop describing GitHub OAuth as a prerequisite for Fly handoff usage.
- **D-11:** Obsolete hosted Auth.js routes/pages should be removed or converted into inert legacy surfaces so the running app no longer exposes a live GitHub auth dependency.

### the agent's Discretion
- The exact launch-claim cookie/session helper shape, as long as it stays server-side and short-lived.
- The exact copy for launch-expired and launch-invalid screens, as long as they are phone-readable and actionable.
- Whether legacy pairing pages remain as compatibility redirects or become explicit legacy notices, as long as they no longer rely on GitHub OAuth.

</decisions>

<specifics>
## Specific Ideas

- The generated handoff URL already exists and is short-lived; the missing piece is the hosted `/launch/[publicId]` consumption flow.
- The current Fly deployment is already failing in practice because the runtime still redirects to `/sign-in` and then dead-ends when OAuth is absent.
- Bridge installations already provide a durable ownership link that can seed device-session issuance without introducing a new account system.
- Existing device sessions can stay 7 days; the main runtime change is removing the short-lived GitHub browser session as a prerequisite.

</specifics>

<deferred>
## Deferred Ideas

- Full database cleanup of legacy `users`, `oauth_accounts`, and `web_sessions` tables.
- A broader multi-bridge identity redesign in relay ownership beyond what is needed for the current handoff path.
- Non-handoff account onboarding or generic unauthenticated session browsing.

</deferred>
